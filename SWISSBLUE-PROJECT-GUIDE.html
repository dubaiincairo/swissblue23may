1 Start here — the 60-second mental model

What this project actually is, in plain language, before any jargon.

This folder is the SwissBlue Hotels website — a real, live, bilingual (Arabic + English) hospitality site for a hotel group operating in Saudi Arabia (Jeddah, Jazan, Riyadh). Internally the project is named wdgroupwebsite28 (WD = Watan Designs Group). It is a Next.js application written in TypeScript, styled with Tailwind CSS, with its editable text and photos stored in a cloud content system called Sanity, and it is published to the internet through Vercel.

The restaurant analogy

If the website were a restaurant, here is who's who. It makes the rest of the guide click into place:

  • The kitchen = Next.js / the server. When a visitor asks for a page, the kitchen cooks it fresh (renders HTML) and sends out a finished plate.
  • The recipe book = your code in src/. The pages and components describe how every dish is made.
  • The pantry = Sanity (the CMS). The ingredients that change often — headlines, descriptions, photos — live here so a non-programmer can restock them without touching the recipe book.
  • The default ingredients = src/lib/content.ts. If the pantry is empty, the kitchen falls back to a built-in set of ingredients shipped with the recipe book, so a plate always goes out.
  • The waiter's order pad = /secretpanel. A custom back-office screen where staff edit the words and pictures. It writes new ingredients into the pantry.
  • The building & utilities = Vercel. The premises that host the kitchen, handle the crowd, and keep the lights on 24/7.
  • The contractor you hire by text message = an AI coding agent. It reads the house rules (AGENTS.md), renovates the recipe book, and hands changes back through Git + Vercel.
Why this design is clever: most of the site is fast, version-controlled code, but the bits that change weekly (a price, a hero photo, a tagline) are editable live by non-coders — and updates appear on the public site within ~5 seconds, with no re-deploy. You get the safety of code and the speed of a CMS.

The four big ideas you'll see everywhere

It renders on the server, fresh, every time

Pages are marked export const dynamic = "force-dynamic". The HTML is built on the server on each request, so the newest content from the pantry always shows. (Many Next.js sites cache pages; this one deliberately doesn't.)

One language tree, mirrored twice

Arabic lives at the root (/, right-to-left) and English is a parallel copy under /en (left-to-right). A small router file decides which one you get.

Defaults in code, overrides in the cloud

Every page asks one function, getEditableContent(), for its words and images. That function merges built-in defaults with whatever edits are saved in Sanity.

Humans and AI edit different layers

Non-coders change content in /secretpanel. Developers (and AI agents) change code & structure via Git. The two rarely collide.

4 How it works as a website

Follow one visitor from typing the address to seeing the page. This is the request lifecycle.

What happens when someone opens the site

The browser asks Vercel for a page

A visitor goes to the site. The request hits the Next.js app running on Vercel's servers.

The router picks a language

src/proxy.ts inspects the URL. Plain / is sent to /en (English). A /ar URL is quietly served the Arabic pages that live at the root. This is how one codebase serves two languages.

The matching page component runs on the server

Next.js finds the file for that route (e.g. src/app/en/page.tsx). Because it's an async Server Component, it can fetch data before sending anything to the browser.

The page asks for its content

It calls getEditableContent(), which reaches into Sanity for the saved overrides and merges them on top of the code defaults, returning one tidy object split into { ar, en }.

Components turn data into HTML

The page hands that data to React components (SiteHeader, HeroMediaCarousel, FaqAccordion…). The rich() helper safely renders formatted text. The server produces finished HTML.

The browser shows it, then "hydrates"

The visitor sees content immediately. Then the interactive bits (carousels, menus) "wake up" in the browser — this is hydration. A few small background helpers also start, including the live-refresh poller (next section).

The Arabic / English routing, concretely

There are two parallel page trees. Anything under src/app/en/ is the English site; everything else directly under src/app/ is the Arabic site. They share the same components and the same content object — they just read the .ar or .en half of it.

// src/proxy.ts — the traffic director (runs before every matched request)
if (path === "/")        → redirect to "/en"   // default visitors get English
if (path === "/ar")      → rewrite  to "/"     // /ar shows the Arabic root page
if (path.startsWith("/ar/")) → strip "/ar" prefix    // /ar/contact → /contact (Arabic)
Redirect vs. rewrite: a redirect changes the address in the visitor's bar (/ visibly becomes /en). A rewrite keeps the address but secretly serves a different file (/ar stays in the bar but renders the root Arabic page). Knowing this difference is a genuinely useful piece of web literacy.

Server Components vs. Client Components — the single most important Next.js idea

Files here are one of two kinds, and you can tell them apart by the first line:

  • Server Component (the default). No special marker at the top. Runs only on the server, can fetch data directly, can't use buttons/clicks/state. Most page.tsx files are these.
  • Client Component. Starts with "use client";. Runs in the browser, can use clicks, typing, timers, and React state. The editor (secret-panel.tsx) and the live-refresh poller are these.
Rule of thumb in this repo: if a file needs interactivity (state, clicks, useEffect), it'll have "use client" at the top. If it just fetches data and lays out HTML, it won't.

5 The live-editing system (the heart of this project)

This is the most original part of the codebase: a custom CMS layer that lets non-coders change the live site in seconds. Worth understanding deeply.

The journey of one edit

Staff open /secretpanel

A client component (secret-panel.tsx, ~1,800 lines) loads. On open it does GET /api/site-content to fetch the current merged content and fills its editing fields.

They edit, with AI helpers on tap

Text is edited in TipTap rich-text boxes. Optional buttons call helper endpoints: Rephrase → Gemini, Translate → DeepL, stock photos → Unsplash/Pexels search, and direct image upload → stored in Sanity.

They hit Save

The panel sends PUT /api/site-content with the whole content object as JSON.

The server writes one document to Sanity

saveEditableContent() calls createOrReplace() on a single fixed document (site-content-singleton) and stamps a new updatedAt timestamp. Everything lives in that one record.

Every visitor's page is quietly watching

LiveContentRefresh (mounted in the root layout) polls GET /api/site-content/version every 5 seconds. It only reads the timestamp.

The live site updates itself

When the timestamp changes, the poller calls router.refresh(). Next.js re-renders the page on the server with the new content and swaps it in — no full reload, no re-deploy. The edit is live in ~5 seconds.

The defaults-plus-overrides merge (why the site never breaks)

The single biggest file in the project, src/lib/editable-content.ts (~2,600 lines), exists to do one job safely: combine the built-in defaults with the saved edits.

// simplified shape of getEditableContent()
const overrides = await sanity.fetch(theSingletonDocument)   // may be empty/null
return mergeContent(overrides)   // defaults underneath, overrides on top

// mergeContent spreads code defaults first, then the saved values:
homepage: { ...defaults.ar.homepage, ...(overrides?.ar?.homepage ?? {}) }
  • If Sanity isn't configured or the fetch fails, it simply returns the code defaults — the site still renders.
  • If a field was never edited, the default shows. If it was edited, the override wins.
  • Image helpers (e.g. syncSharedImages, syncHeroSlides) keep photos consistent across pages after edits.
This is a resilience pattern worth stealing. "Ship sensible defaults in code, allow overrides from a database, merge at read time." It means a misconfigured CMS or a network blip degrades to a working site instead of a blank one.

Two editing doors

The project actually has two ways to edit content, and it helps not to confuse them:

  • /secretpanel — the custom-built, friendly editor tailored to this site's exact fields, with the AI rephrase/translate/photo tools. This is the everyday door.
  • /studio — the full standard Sanity Studio dashboard (mounted by src/app/studio/[[...tool]]/page.tsx). More raw and general-purpose; the power-user door.

6 How agentic AI develops & communicates with this repo

You asked specifically about this. Here's how an AI coding agent (like the one reading your project) actually plugs into a codebase like this one.

The agent's "house rules" files

An AI coding agent doesn't guess how to behave — it reads instruction files committed to the repo. This project has them, and they're short and powerful:

  • CLAUDE.md — the entry point an agent looks for. Here it contains a single line, @AGENTS.md, which means "import the rules from AGENTS.md." (The @file syntax pulls in another file's content.)
  • AGENTS.md — the real rulebook. It tells the agent two critical things, quoted from your repo:
// 1) Don't trust your memory of this framework
"This is NOT the Next.js you know … Read the relevant guide in
 node_modules/next/dist/docs/ before writing any code."

// 2) The exact deployment workflow
- branch staging  → auto-deploys to a preview URL
- branch main     → production
- Finish work on a claude/* branch, open a PR into staging, merge it
  WITHOUT asking. NEVER push to main unless the user says "deploy to production".
Takeaway for you: a well-run vibe-coding project keeps a short AGENTS.md (or CLAUDE.md) at its root. It's the single highest-leverage file you can write — it turns vague requests into safe, repeatable behavior. More on this in the next section.

The .claude/ folder — the agent's local settings

  • settings.local.json — a permissions allow-list. It pre-approves specific safe actions (here: npm install, and a preview tool) so the agent isn't stopped to ask about routine commands.
  • launch.json — defines the "web" run configuration (npm run dev on port 3000), so the agent and tools know how to start the site locally.

How the agent actually "communicates" with the project

Reads

It opens files (like this guide did), greps for symbols, and reads the instruction files to load context.

Edits

It makes precise changes to .ts/.tsx files on a feature branch named claude/....

Runs commands

Through a shell it runs npm install, npm run build, npm run lint, and Git commands to commit and push.

Opens a Pull Request → staging

Following AGENTS.md, it merges into the staging branch. Vercel sees the push and auto-builds a preview you can click and review.

Waits for your word on production

It will not touch main (production) until you explicitly say so. Your standing approval covers staging only.

Where AI must be careful here: secrets live in .env.local (Sanity write token, Gemini/DeepL/Unsplash keys). That file is .gitignored and must never be printed, committed, or leaked. A good agent treats it as read-only and invisible.

7 Vibe coding, professionally

You called yourself a vibe coder and asked how to do it more professionally. Here's a practical playbook, anchored to things you can see in this repo.

What "vibe coding" is — and its one real risk

Vibe coding means building by describing what you want to an AI and steering it, rather than typing every line. It's legitimate and fast. The single risk is shipping code you don't understand, which bites you when something breaks at 11pm and you can't tell the AI where to look. Everything below is about closing that gap without slowing down.

Eight habits that turn vibe coding into engineering

Keep an AGENTS.md and treat it as living law

Yours already encodes the deploy rules. Add to it: coding conventions, "never do X," where things live. Every hour you invest here is repaid on every future prompt.

Branch, preview, then promote

Your staging → main split is exactly right. Never let AI (or yourself) edit production directly. Always click the staging preview before promoting.

Commit in small, labelled steps

Many small commits with clear messages give you an "undo" button at every point. If a change breaks the build, you can revert just that step.

Make the AI explain before it edits

Ask "what files will you touch and why?" first. If the plan is wrong, you've saved a mess. This guide's tree is a map you can use to sanity-check those answers.

Read the diff, not just the result

Before merging, skim what changed. You don't need to understand every line — just confirm it touched the files you expected and nothing in .env.local or unrelated areas.

Let the tools catch mistakes

Run npm run lint and npm run build before deploying. TypeScript + ESLint here will flag whole classes of bugs for free. Green build = safe to ship.

Guard your secrets

API keys belong only in .env.local (gitignored) and in Vercel's dashboard. Never paste them into chat, code, or a public file.

Learn one concept per task

Each time the AI does something, pick one term from the glossary below and actually understand it. In a month you'll read your own codebase fluently.

The vocabulary that makes you sound (and think) like a developer

When you can say these to an AI, you get far better results: componentpropsstate server vs clientrouteAPI endpoint environment variablebranchpull request builddeploymerge / rebase refactorregression
